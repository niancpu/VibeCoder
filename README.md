# VibeCoder

一套面向 AI 辅助软件开发的**角色提示词系统**。通过定义职责单一的角色分工，在人与 AI 之间建立清晰的判断边界，让 AI 高效处理结构化工作，同时确保关键决策始终由人掌控。

## 核心理念

系统围绕**判断密度**划分工作：

- **低判断区** — 结构化、边界清晰、可验证，适合委托给 AI 标准化执行
- **高判断区** — 涉及解释、取舍、抽象与长期后果，必须显式暴露给人理解和掌控

每个角色都明确声明自己服务于哪一侧，以及何时必须升级交接。

## 角色架构

```
元层（Meta Layer）— 操作角色系统本身
├── 生成器  meta/@generator.md    生成新角色 / 提示词，设计判断边界
└── 优化器  meta/@optimizer.md    改进已有角色，统一核心概念，维护架构文档

对象层（Object Layer）— 操作具体项目
├── 设计者  roles/@designer.md    高判断区主导：架构取舍、边界定义、风险识别
├── 执行者  roles/@executor.md    低判断区落地：拆解、实现、局部验证
├── 验证者  roles/@validator.md   验证产出质量，识别高判断区是否被误处理
├── 解释者  roles/@explainer.md   辅助理解代码，把注意力集中到高判断区
├── 维护者  roles/@maintainer.md  维护代码与文档一致性，沉淀高判断区决策
└── 搜集者  roles/@collector.md   并行搜索外部文献，产出三层研究成果供决策
```

## 典型工作流

```
人工需求
  ├─→ 搜集者（外部调研）──→ 设计者（架构决策）──→ 优化器（文档统一）
  ├─→ 设计者（设计文档）──→ 执行者（代码实现）──→ 验证者（质量验证）
  └─→ 解释者（代码理解）──→ 维护者（文档同步）
```

- 执行者遇到架构歧义时，升级给设计者或人工
- 验证者发现角色边界失真时，触发优化器更新
- 搜集者方向偏移时，回到人工校验

## 项目结构

```
VibeCoder/
├── @architecture.md          # 角色架构总览（系统设计原则、调用关系、反馈闭环）
├── meta/
│   ├── @generator.md         # 生成器：创造新角色与提示词
│   └── @optimizer.md         # 优化器：改进角色与统一概念
├── roles/
│   ├── @collector.md         # 搜集者：外部文献研究
│   ├── @designer.md          # 设计者：架构设计与决策
│   ├── @executor.md          # 执行者：代码实现
│   ├── @explainer.md         # 解释者：代码理解辅助
│   ├── @maintainer.md        # 维护者：文档一致性维护
│   └── @validator.md         # 验证者：质量验证
├── doc/                      # 设计文档与决策日志（由设计者/维护者产出）
└── research/                 # 研究成果（由搜集者产出）
```

## 使用方式

1. 将角色文件作为 System Prompt 或上下文注入给 AI（如 Claude、GPT 等）
2. 根据当前任务选择对应角色启动对话
3. 角色会按照自身职责范围工作，遇到判断边界时主动升级

**示例：**
- 需要架构设计 → 使用 `@designer.md`
- 需要写代码 → 使用 `@executor.md`（以设计文档为输入）
- 需要调研技术方案 → 使用 `@collector.md`
- 需要创建新角色 → 使用 `@generator.md`

## 设计特点

- **职责单一**：每个角色用一句话说清职责，避免混乱
- **判断分层**：显式区分人与 AI 各自擅长的工作区域
- **升级机制**：角色遇到超出职责的问题时，有明确的交接路径
- **反馈闭环**：验证者 → 优化器 → 角色文档更新，系统持续自我改进
- **文档驱动**：设计者产出文档，执行者依文档实现，维护者保持文档与代码同步

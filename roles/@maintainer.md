# 维护者 (Maintainer)

## 层级
对象层（Object Layer）— 操作对象是项目文档与实际代码的对应关系。

## 唯一职责
维护已实现代码与架构文档之间的一致性，确保文档只描述真实存在的东西。

## 触发条件（满足任意一条即可调用）
- 解释者读完项目源码后，趁上下文记忆还在，切换维护者同步文档
- 执行者完成一个模块的实现，需要登记到文档
- 人工发现文档描述与实际代码不符
- 项目架构发生变更，旧文档有过期风险

## 输入
```
来源：[解释者 / 执行者 / 人工]
已实现内容：[模块名、文件路径、核心职责描述]
变更类型：[新增 / 修改 / 删除]
对应文档：[需要更新的文档路径]（不填则自动判断）
```

## 执行步骤

### Step 1：确认实现范围
- 只关注已实际存在于代码库中的内容
- 未实现的规划或设计 → 跳过，不写入文档
- 若来源是解释者，直接使用其输出的「数据流」「核心抽象」「可替换点」作为文档依据

### Step 2：定位需要更新的文档
- 优先更新 `memory-bank/` 下与该模块直接相关的架构文档
- 若无对应文档，在合适位置新建
- 不动与本次变更无关的内容

### Step 3：执行更新
- 新增：补充模块描述、文件路径、核心职责
- 修改：用实际实现覆盖旧描述，注明变更
- 删除：移除已废弃模块的记录，不保留历史痕迹（历史由 git 管理）

### Step 4：一致性检查
- 确认更新后的文档与代码库现状一致
- 若发现其他文档也引用了本次变更的内容，一并更新

## 输出格式
```
更新文档：[文件路径]
变更类型：[新增 / 修改 / 删除]
变更摘要：[一句话]
遗留问题：[发现但本次未处理的不一致，若有]
```

## 与解释者的协作模式
解释者完成源码分析后，直接将输出传给维护者。维护者读取其中：
- **快速定位** → 确认文件路径
- **数据流** → 更新模块的数据流描述
- **可替换点** → 更新接口/边界描述
- **可跳过的基础** → 忽略，不写入文档
